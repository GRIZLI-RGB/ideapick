# Ideapick — backlog (нереализованные фичи)

Фичи и продуктовые гипотезы, которые **ещё не в MVP**, но стоит реализовать или проработать отдельно.  
Текущая спецификация продукта — в [`docs/spec.md`](./spec.md).

---

## 1. Следующие шаги после анализа (upsell)

**Задача:** повысить конверсию — продавать не только оценку идеи, но и **следующий практический шаг** для тех, кто решил «браться».

**Когда показывать:** на `/app/ideas/[id]` под готовым отчётом анализа — отдельный блок с 1–2 CTA (не до анализа).

| Продукт | Цена | Что получает пользователь |
|---------|------|---------------------------|
| **План запуска на 30 дней** | 149 ₽ | Пошаговый план: что делать по неделям/дням, приоритеты, метрики проверки гипотез |
| **ТЗ для MVP** | 99 ₽ | Техническое задание: scope MVP, ключевые экраны/фичи, стек, риски и оценка сложности |

**Вход для AI:** название + описание идеи + снимок отчёта анализа на момент заказа.

**Механика:** как у анализа — цена + confirm, hold → AI → списание или возврат при ошибке. Каждый продукт заказывается отдельно; повторный заказ **заменяет** предыдущий результат того же типа (история версий не хранится).

**Статус:** кандидат на первый слой монетизации после базового анализа. Требует проработки промптов, UI блока upsell и схемы хранения результатов.

## 2. Шаблоны описания идей

**Задача:** снизить порог входа при создании «своей» идеи и дать AI более структурированный контекст для анализа.

**Статус: реализовано** в `components/ideas/add-idea-dialogs.tsx` — раскрывающийся блок «Начать с шаблона» в диалоге «Своя идея»; 6 типов (SaaS, маркетплейс, мобильное приложение, курс/инфопродукт, физический товар, услуга/агентство). Выбран вариант с предзаполнением текста (не placeholder); затирание чужого текста — через подтверждение повторным кликом.

**Осталось из открытых вопросов:** сохранять ли выбранный тип шаблона в метаданных идеи (сейчас не сохраняется).

## 3. История версий анализа

**Задача:** не терять контекст при пересчёте — показывать, как менялась оценка идеи со временем.

**Сейчас в spec:** при «Обновить анализ» предыдущий отчёт полностью заменяется, история не хранится.

**Целевое поведение:**

- при каждом обновлении анализа старая версия **сохраняется** (версия 1, версия 2, …);
- на странице идеи — переключение между версиями или список с датами;
- **график изменения score** (и, возможно, ключевых метрик из отчёта) по версиям.

**Открытые вопросы:** лимит хранимых версий; влияет ли на ценообразование (99 ₽ за каждую версию как сейчас); что показывать в списке идей — только последний score или дельту.

## 4. Слайды для инвестора (и других аудиторий)

**Задача:** монетизировать готовый отчёт — превратить анализ в артефакт для презентации.

**Когда показывать:** после оплаченного анализа — кнопка вроде «Сделать слайды для инвестора · 199 ₽» (возможны варианты аудитории: инвестор, партнёр, команда — отдельные пресеты или один универсальный набор).

**Что получает пользователь:** AI на основе отчёта анализа генерирует текст для **5 слайдов**:

1. Проблема  
2. Решение  
3. Рынок  
4. Бизнес-модель  
5. Дорожная карта  

**Вход для AI:** название + описание идеи + снимок отчёта анализа на момент заказа.

**Механика:** как у анализа — цена + confirm, hold → AI → списание или возврат. Формат выдачи (markdown, PDF, экспорт в Google Slides) — TBD.

**Статус:** идея upsell; пересекается с п. 1 — стоит продумать единый блок «следующие шаги» на странице идеи.

## 5. Лендинг: «Почему не ChatGPT?»

**Задача:** снять возражение «зачем платить, если можно спросить ChatGPT» — короткий блок на лендинге.

**Суть месседжа (черновик):**
- ChatGPT даёт общий текст без фиксированной структуры и вердикта
- Ideapick — отчёт из 6 разделов, оценка 0–100, первые шаги под соло-разработчика
- Платите только за анализ (99 ₽), без подписки; первый — за счёт бонуса при регистрации

**Где:** отдельная секция или подблок в FAQ / «Как это работает».

**Статус:** после релиза; для MVP не блокер.

## 6. Лендинг: social proof

**Задача:** повысить доверие отзывами, цифрами («N анализов»), логотипами — когда появятся реальные данные.

**Принцип:** не показывать накрученные или вымышленные отзывы — хуже, чем отсутствие блока (избитый паттерн, снижает доверие).

**Когда делать:** после появления реальных пользователей и метрик, которые можно честно показать.

**Варианты:**
- Счётчик анализов / регистраций (если цифра уже внушает)
- 1–3 коротких отзыва с именем и ролью (разработчик, фаундер)
- Цитаты из поддержки / твиттер (с согласия)

**Статус:** отложено до post-launch.

## 7. Уведомления по тикетам поддержки

**Задача:** не заставлять пользователя постоянно заходить в приложение, чтобы узнать, ответила ли поддержка — при новом ответе на тикет отправлять письмо на email аккаунта.

**Когда отправлять:** при добавлении сообщения с автором `support` в тикет пользователя (не при его собственных сообщениях и не при закрытии без нового ответа).

**Что в письме (черновик):**
- тема тикета;
- превью текста ответа (или полный текст, если короткий);
- ссылка на тикет в `/app/support` (или прямо на конкретный тикет).

**Технически:** переиспользовать существующую почтовую инфраструктуру (как для magic link); шаблон в стиле продукта.

**Открытые вопросы:** отправлять ли уведомление при смене статуса без нового сообщения; нужен ли opt-out в настройках; дебаунс при нескольких ответах подряд.

**Статус:** тикеты поддержки есть; email-уведомления при ответах — не реализованы.
