## Задача анализа

Помочь пользователю **рационально оценить идею** по названию и описанию: спрос, конкуренция, монетизация, сложность, риски и другие факторы — и свести их к **одному рейтингу 0–100** («стоит ли игра свеч»).

Анализ — **рекомендация AI**, не инвестиционный совет и не гарантия результата.

**Аудитория:** русскоязычный СНГ, в первую очередь разработчики и соло-предприниматели.  
**География оценки:** ориентир на СНГ-рынок, если из текста идеи не следует иное.

---

## Вход

| Поле | Ограничения |
|------|-------------|
| **Название** | 5–100 символов после trim |
| **Описание** | минимум 30 символов после trim |

Название и описание **не редактируются** после создания идеи. На анализ уходит снимок текста на момент запуска.

### Зачем такие минимумы

**Hard limits** (проверка при создании идеи и повторно перед анализом):

- **5 символов в названии** — отсекает одну букву или пустую строку, но не запрещает короткий бренд («Lunacy», «Bebra»).
- **30 символов в описании** — хотя бы одно осмысленное предложение: кому, что, зачем.

**Смысловая проверка** — не hard limit, а **AI-превалидация** перед платным анализом (см. ниже):

- «Кот», «Игра», «Идея» без контекста → низкое качество, предупреждение.
- Короткое кодовое название («Lunacy») **допустимо**, если описание раскрывает продукт, аудиторию и суть.
- Хороший паттерн названия: *что это* в нескольких словах — «Бот для авто-выдачи игровых аккаунтов», «Сервис адаптации сайтов для слабовидящих».

### Почему без поля «Тип проекта»

Отдельный селект («Сайт», «Офлайн-бизнес», «SaaS»…) чуть усложняет UI и дублирует то, что уже должно быть в описании. Тип проекта AI **выводит из текста** при анализе; пользователь не выбирает его вручную.

## Превалидация (перед анализом)

Двухшаговая схема. **Запуск анализа не блокируется** — только предупреждения.

### Шаг 1 — Hard limits

Если не проходит — кнопка «Запустить анализ» **неактивна**, показать причину (не хватает символов в названии/описании).

### Шаг 2 — AI quality check

Дешёвый вызов модели **до** списания 99 ₽. Вход: title + description. Выход — JSON:

```json
{
  "quality": 0,
  "issues": ["строка"],
  "hints": ["строка"]
}
```

| quality | Поведение UI |
|---------|----------------|
| **≥ 60** | Без предупреждения, сразу confirm оплаты |
| **40–59** | Confirm + текст: «Описание скудное — оценка может быть неточной. Лучше дописать детали (можно создать новую идею).» + кнопки «Всё равно запустить» / «Отмена» |
| **< 40** | Явное предупреждение: «Данных мало для осмысленного анализа; результат будет очень приблизительным.» + те же кнопки |

`issues` — что не так (расплывчато, нет аудитории, нет monetization hint).  
`hints` — 1–2 совета, что дописать (без редактирования текущей идеи — «создайте новую с уточнённым описанием»).

Pre-check **не списывает** 99 ₽. Списание — только после confirm и успешного полного анализа.

## Пайплайн анализа

```
title + description
  → hard limits
  → AI prevalidation (quality)
  → [confirm + hold 99 ₽]
  → полный AI-анализ (+ web search — TODO, см. ниже)
  → JSON-отчёт + score
  → списание / возврат hold при ошибке
```

**Первый запуск:** кнопка «Запустить анализ · 99 ₽».  
**Обновление:** «Обновить анализ · 99 ₽» — тот же пайплайн, тот же текст идеи; отчёт и score **полностью заменяются**; на UI — дата последнего анализа. История версий не хранится.

Обновление имеет смысл, когда изменился **внешний контекст** (рынок, конкуренты, регуляторика), а формулировка идеи осталась прежней.

## Выход: score

Одна итоговая оценка **0–100** — синтез всех факторов отчёта. Под-оценки по осям **нет** (расшифровка при наведении — возможно позже).

Модель должна оценивать **реалистично**, без системного завышения или занижения.

### Диапазоны (UI)

Согласованы с `lib/ideas/score-style.ts`:

| Score | Смысл (черновик) | Цвет в UI |
|-------|------------------|-----------|
| **71–100** | Сильная идея при типичных ограничениях соло | зелёный |
| **51–70** | Имеет смысл проверять, есть оговорки | янтарный |
| **31–50** | Сомнительно, нужна сильная дифференциация | голубой |
| **0–30** | Слабая привлекательность / высокие риски | розовый |

Границы и формулировки — **черновик**, подгоним по реальным отчётам.

Планируем **минимум текста, максимум пользы**: короткие списки, без простыней.

Кандидаты на секции (не утверждено):

- краткий вывод (2–3 предложения);
- спрос / аудитория;
- конкуренция;
- монетизация;
- сложность для одного человека;
- риски;
- первые шаги проверки гипотезы.

При выборе секций полезно опираться на **метрики рынка** (TAM / SAM / SOM) — см. [Приложение A](#приложение-a-tam-sam-som-и-смежные-метрики).

### Черновик корневого JSON (без секций)

```json
{
  "score": 0,
  "analyzedAt": "ISO-8601",
  "sections": {}
}
```

## Приложение A. TAM, SAM, SOM и смежные метрики

Краткий справочник для обсуждения секций отчёта. **Не путать** метрики **рынка** (сколько денег/клиентов существует) с метриками **продукта** (сколько людей уже пользуется вашим сервисом).

### TAM / SAM / SOM — «воронка рынка»

Оценивают **размер возможности**, обычно в деньгах или количестве потенциальных клиентов.

| Метрика | Расшифровка | Пример для «SaaS учёта подписок для фрилансеров в РФ» |
|---------|-------------|--------------------------------------------------------|
| **TAM** (Total Addressable Market) | Весь теоретический рынок, если бы продуктом мог пользоваться **каждый** в категории | Все самозанятые и ИП в РФ, которым вообще нужен учёт денег |
| **SAM** (Serviceable Addressable Market) | Часть TAM, которую вы **реально можете обслужить** продуктом и каналами (география, язык, формат) | Русскоязычные соло-разработчики и дизайнеры на НПД, готовые платить за SaaS |
| **SOM** (Serviceable Obtainable Market) | Доля SAM, которую **реалистично захватить** за 1–3 года с вашими ресурсами | 0,1–2% SAM — первые платящие через контент, Telegram, сарафан |

**Зачем в Ideapick:** AI не обязан считать TAM/SAM/SOM в рублях с точностью до миллиона — но **логика** «рынок огромный / узкая ниша / достать можно только маленький кусок» напрямую влияет на score и блок про спрос.

### MAU / DAU / WAU — метрики продукта (не рынка)

| Метрика | Расшифровка |
|---------|-------------|
| **DAU** | Daily Active Users — уникальные пользователи за день |
| **WAU** | Weekly Active Users |
| **MAU** | Monthly Active Users |

Это про **ваш** сервис после запуска, не про оценку идеи до запуска. В анализе могут упоминаться как «какие метрики успеха отслеживать на MVP», но это не SAU/SAM.

*(SAU как отдельной стандартной метрики нет — возможно, имелись в виду SAM или MAU.)*

### Смежные понятия (часто рядом в отчётах)

| Понятие | Суть |
|---------|------|
| **Product-market fit** | Есть повторяющийся спрос, пользователи возвращаются и платят |
| **Конкуренция / насыщенность** | Сколько похожих решений, барьер входа |
| **Willingness to pay** | Готовность платить у целевой аудитории |
| **CAC / LTV** | Стоимость привлечения клиента vs его lifetime value — для зрелой стадии; на этапе идеи — качественно («CAC будет высоким, если…») |
| **Unit-экономика** | Сходится ли модель на одном клиенте / одной продаже |

### Как это ложится на Ideapick

Для секции «спрос / рынок» AI может давать **качественную** оценку в духе TAM → SAM → SOM (без выдуманных цифр «рынок = 847 млрд ₽»). Если данных мало — явно писать «оценка грубая, опирается только на описание».
